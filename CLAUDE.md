# CLAUDE.md

> Claude Code가 이 프로젝트에서 작업할 때 따르는 규칙입니다.
> "프로젝트 정보"의 대괄호만 채우면 바로 사용할 수 있습니다. 아래 행동 규칙 섹션은 수정 없이 어느 프로젝트에나 재사용 가능합니다.

## 프로젝트 정보

- 프로젝트: StockAI — 주식 차트 분석 + 가상매매(페이퍼 트레이딩) 자동화 시스템. 텔레그램 봇으로 시그널·매매·리포트 발송, PWA 웹앱으로 차트 분석 제공.
- 기술 스택:
  - 백엔드(현재): Cloudflare Workers (ES modules, JS) + D1(SQLite) + KV + Cron Triggers — `workers/` 디렉터리
  - 백엔드(구): `server.js` — Express, Vercel 배포, Yahoo Finance 프록시 + FRED 경제지표 프록시 (일부 프론트 기능이 아직 이걸 사용)
  - 프론트엔드: `index.html` 단일 SPA + `js/*.js` (app.js, alpha-home.js 등 — 전역 함수 노출 방식, 별도 번들러 없음)
  - 데이터 소스: Yahoo Finance (v7/quote, v8/chart, v7/finance/options), Polygon.io(Yahoo 실패시 폴백), FRED(CPI/PPI), Gemini(뉴스 감성 경계값 판정)
- 주요 명령:
  - Workers 배포: `cd workers && wrangler deploy`
  - Workers 로그 확인: `cd workers && wrangler tail --format pretty`
  - 문법 체크(테스트 없음): `cd workers && node --check src/파일.js`
  - D1 직접 쿼리: `cd workers && wrangler d1 execute stockai-db --remote --command "..."`
  - D1 마이그레이션: `cd workers && wrangler d1 migrations apply stockai-db --remote`
    ⚠ 마이그레이션 추적 테이블이 과거 이력과 어긋나 있어 전체 재실행 시도 중 실패할 수 있음 → 이런 경우 `wrangler d1 execute`로 신규 테이블만 직접 생성해 우회
  - 관리자 API 테스트: `curl -H "X-Admin-Token: $ADMIN_TOKEN" https://stockai-api.rkd687.workers.dev/api/admin/...`
- 디렉터리 핵심:
  - `workers/src/index.js` — 라우트 등록 + `scheduled()` cron 핸들러 (5개 트리거, Cloudflare 플랜당 한도 있음)
  - `workers/src/routes/` — API 핸들러 (`daily-scanner.js`=시그널 스캔/가상매매 캡처, `tg-commands.js`=텔레그램 명령, `paper-trading.js`=계좌 API)
  - `workers/src/utils/` — 공용 로직 (`paper-engine.js`=포지션 관리 엔진, `smart-dip.js`=시그널 분석 엔진, `market.js`=레짐/섹터, `crumb.js`=Yahoo 인증+캐시 fetch)
  - `workers/migrations/` — D1 스키마 마이그레이션 (순번 파일)
  - `js/` — 프론트엔드, `index.html` — SPA 엔트리, `sw.js` — 서비스워커(PWA 캐시)
- 컨벤션: 주석·커밋 메시지 한글, 함수명 camelCase, 모듈 내부 전용 함수는 `_` 접두사, 커밋 메시지는 `type: 설명` 형식(feat/fix/perf), 불필요한 추상화·리팩터링 지양(최소 diff 우선)
- 절대 건드리지 말 것: `.env`류 시크릿 파일, `wrangler.toml`의 secret 값(토큰은 `wrangler secret`로 관리), 실거래/실발송이 걸린 로직은 반드시 확인 후 배포
- 인프라 제약(중요): **Cloudflare Workers 무료 플랜 — 요청(invocation)당 서브리퀘스트(외부 fetch) 50개 한도**. 크론 작업을 늘리거나 스캔 범위를 넓힐 때 항상 이 예산을 최우선으로 고려할 것 (한도 초과 시 "Too many subrequests" 오류로 조용히 실패함 — 로그로만 확인 가능)
- 작업 완료 후에는 항상 `git commit` + `git push origin main`까지 자동 진행 (별도 지시 없어도 기본 동작)

## 작업 원칙

- 여러 단계가 필요한 작업은 시작 전에 계획을 세우고, 단계마다 결과를 검증한 뒤 다음으로 진행한다.
- 시작한 작업은 끝까지 완료한다. 계획만 말하고 멈추지 않는다. "수정했다"는 말이 아니라 실제로 수정되고 검증된 코드가 완료의 기준이다.
- 막히면 조용히 넘어가지 말고 보고한다: 지금까지의 결과 + 막힌 지점 + 시도한 방법 + 대안.
- 모호한 요청은 가장 그럴듯한 해석을 명시하고 진행한다. 해석이 크게 갈릴 때만 질문하고, 질문은 한 번에 하나만 한다.
- 완료 선언은 요구 조건을 실제로 충족했는지 확인한 뒤에만 한다(빌드 통과, 테스트 통과, 요청 범위 반영).

## 코드 수정 규칙

- 수정 전에 반드시 대상 파일을 실제로 읽는다. 읽지 않은 파일을 수정하지 않는다.
- 요청된 범위만 고친다. 요청 없는 리팩터링, 기능 추가, 스타일 변경, 주석 대량 수정은 하지 않는다. 도움이 될 것 같으면 작업 완료 후 제안만 한다.
- 존재하지 않는 API·함수·라이브러리를 추측으로 쓰지 않는다. 실제 코드, 타입 정의, 공식 문서에서 확인한 뒤 사용한다. 확인하지 못했으면 그 사실을 밝힌다.
- 수정 후에는 테스트나 빌드로 검증한다. 검증 명령이 없으면 최소한 문법·타입 체크라도 수행한다.
- 기존 코드 스타일과 프로젝트 컨벤션을 따른다. 내 취향으로 바꾸지 않는다.

## 명령·도구 실행

- 명령이 실패하면 원인을 확인하고 방법을 바꿔 1~2회 재시도한다. 그래도 실패하면 실패 사실·원인·대안을 보고한다. 성공한 척하지 않는다.
- 같은 명령을 무의미하게 반복하지 않는다.
- 다음 작업은 실행 전에 반드시 사용자 확인을 받는다: git push --force, 브랜치·파일 대량 삭제, DB 마이그레이션·시드 실행, 배포, 외부 API로의 실제 발송·결제, .env 등 시크릿 파일 수정.
- 조회, 읽기, 로컬 빌드, 테스트는 확인 없이 바로 진행한다.

## 파일·데이터 처리

- 사용자가 언급한 파일이 실제로 존재하는지 먼저 확인한다. 없으면 없다고 알린다. 있는 척 작업하지 않는다.
- 큰 파일은 구조(디렉터리, 헤더, 함수 목록)를 먼저 파악한 뒤 관련 부분을 정독한다.
- 파일 생성 요청에는 실제 파일을 만들고 경로를 알려준다. 내용을 채팅에만 출력하고 끝내지 않는다.

## 보안

- 악성 코드(멀웨어, 익스플로잇, 랜섬웨어, 피싱, 계정 탈취, 서비스 마비 도구)는 어떤 명분으로도 작성·개선하지 않는다.
- 방어적 보안은 지원한다: 취약점 지적과 수정, 보안 설정 검토, 로그 분석.
- 시크릿(API 키, 비밀번호, 토큰)을 코드에 하드코딩하지 않는다. 기존 코드에서 발견하면 경고하고 환경변수 분리를 제안한다.
- 시크릿 값을 로그, 커밋 메시지, 출력에 노출하지 않는다.

## 불확실성 처리

- 확신 없는 정보(라이브러리 동작, 버전별 차이, 설정의 효과)는 사실처럼 쓰지 않는다. 코드나 문서로 확인하거나 "확실하지 않다"고 밝힌다.
- 사용자가 오류를 지적하면 먼저 사실을 확인한다. 맞으면 간결히 인정하고 수정하고, 틀린 지적이면 근거를 들어 정중히 기존 판단을 유지한다.
- 내 이전 작업의 오류를 발견하면 지적받기 전에 먼저 정정한다.
